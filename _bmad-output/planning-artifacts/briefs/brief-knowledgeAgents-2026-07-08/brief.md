# CloudBrief 选择性引入 GraphRAG 产品简报

## 执行摘要

CloudBrief 当前已具备一套成熟的语义检索链路（BM25 + 向量混合检索 → RRF → Rerank → 引用生成），能够高效回答「某段内容讲了什么」「哪段最相关」这类事实性问题。然而，当用户问题涉及跨文档的实体关系、因果链或比较推理时，现有链路容易出现「找得到片段，拼不出关系」的瓶颈。

本简报建议在 CloudBrief 中**按知识库选择性引入 GraphRAG**：不替换现有检索链路，而是新增一条可选的关系推理分支。首批试点聚焦「实体关系密集」的知识库（如组织架构、高管履历、供应链文档），其余知识库继续沿用向量 RAG。预期在控制成本的前提下，将关系类问题的回答准确率与可解释性提升到可演示级别，为 CloudBrief 从「信息检索助手」向「企业知识推理助手」演进奠定基础。

## 问题背景

### 1. 当前系统能力边界

CloudBrief 的检索链路擅长处理以下问题：
- 退货政策是什么？
- 某产品的 API 如何调用？
- 某篇文档的核心结论是什么？

这些问题有一个共同特征：**答案通常集中在单个或少数几个文本片段中**，语义相似度检索即可召回。

### 2. 未被满足的关系型问题

企业用户在真实场景中还会提出：
- 「张三和李四是否在同一家子公司任过职？」
- 「这个供应商的上游合作伙伴有哪些？」
- 「A 项目延期是否影响了 B 项目的资源投入？」

这类问题的特点是：
- **答案分散**：信息来自多个文档或段落；
- **关系关键**：需要先识别实体，再理解实体间的关联；
- **可解释性要求高**：用户需要看到「A→B→C」的推理路径，而不是一段相关性较高的原文。

现有向量 RAG 对这类问题只能做到语义近似匹配，无法保证关系推理的完整性。

## 解决方案

### 核心思路

在 CloudBrief 中新增 **GraphRAG 可选分支**，按知识库维度决定是否启用。系统架构变为：

```
用户问题
   ↓
查询意图识别 / 路由
   ↓
├─→ 事实性/段落性问题 → 现有混合检索 + Rerank（向量 RAG）
└─→ 关系型/推理性问题 → GraphRAG（实体识别 → 子图检索 → LLM 生成）
   ↓
统一生成与引用展示
```

### 关键设计原则

| 原则 | 说明 |
|---|---|
| **可选而非默认** | 每个知识库独立配置是否启用 GraphRAG，避免一刀切的成本浪费。 |
| **不破坏现有链路** | 现有 BM25 + 向量 + Rerank 链路保持为默认路径，GraphRAG 作为补充。 |
| **按知识库隔离 schema** | 不同知识库可配置不同的实体/关系 schema，避免「一个图谱包打天下」。 |
| **可回退** | 当图谱质量不足或查询无法映射到图时，自动回退到向量 RAG。 |

## 适用范围：按知识库选择性启用

不是所有文档都需要 GraphRAG。建议根据知识库的内容特征和用户查询模式做选择。

### 建议启用 GraphRAG 的知识库

| 类型 | 示例 | 原因 |
|---|---|---|
| 企业关系类 | 组织架构、高管履历、股权结构 | 实体关系密集，跨文档推理需求高 |
| 供应链/合作网络 | 供应商档案、合作协议、采购记录 | 需要回答「上下游」「合作关系」等问题 |
| 项目/事件脉络 | 项目文档、会议纪要、变更记录 | 时间线和因果链是关键 |
| 合规/审计资料 | 审计报告、合规检查清单 | 需要可追溯的关系路径 |

### 不建议启用 GraphRAG 的知识库

| 类型 | 示例 | 原因 |
|---|---|---|
| FAQ / 帮助文档 | 产品 FAQ、操作手册 | 答案是段落级事实，向量 RAG 已足够 |
| 长篇文章/新闻稿 | 行业分析、公司新闻 | 主要是语义总结，关系抽取价值低 |
| 更新极频繁的文档 | 实时数据报表、日报 | 图谱维护成本过高 |
| 非结构化闲聊数据 | 聊天记录、邮件归档 | 关系噪声大，抽取质量不稳定 |

## 与 Advanced RAG 的关系

GraphRAG 不是 Advanced RAG 的对立面，而是 Advanced RAG 工具箱中的一种专门手段。

| 维度 | Advanced RAG（广义） | GraphRAG（特指） |
|---|---|---|
| 定位 | 检索增强的整套优化方法 | 解决关系推理问题的专门架构 |
| 核心数据 | 文本块 + 向量 | 实体 + 关系 + 属性 |
| 检索逻辑 | 语义相似度匹配 | 图遍历 + 关系推理 |
| 适用问题 | 「这段讲了什么？」 | 「A 和 B 有什么关系？」 |

CloudBrief 当前已落地多种 Advanced RAG 技术（混合检索、Rerank、查询改写、引用解析），引入 GraphRAG 是向其能力边界的自然延伸，而非推倒重来。

## 预期成本与风险

### 成本

| 类别 | 主要投入 | 估算级别 |
|---|---|---|
| 研发成本 | 实体/关系抽取、图数据库集成、查询路由、结果融合 | 中-高（2–4 周 PoC，6–10 周 MVP） |
| 计算成本 | 图谱构建时的 LLM 抽取调用、图库存储与查询 | 中 |
| 运维成本 | schema 维护、图谱增量更新、质量监控 | 中 |
| 人力成本 | 需要熟悉图谱工程或愿意学习的工程师 | 中 |

### 风险

| 风险 | 说明 | 缓解措施 |
|---|---|---|
| 抽取质量不稳定 | 实体消歧、关系抽取容易出错 | 限定 schema，先在小范围知识库验证 |
| 图谱维护负担 | 文档更新后需同步更新图谱 | 设计增量更新机制，非全量重建 |
| 查询路由错误 | 关系型问题被路由到向量 RAG，反之亦然 | 初期允许用户/管理员反馈，持续优化路由 |
| ROI 不达预期 | 关系型问题占比可能被高估 | 先通过日志分析确认真实问题分布 |

## ROI 评估

### 收益

- **回答质量提升**：关系型问题从「可能遗漏关键关联」提升至「可给出结构化关系路径」。
- **可解释性增强**：答案附带图遍历路径，便于审计与信任建立。
- **产品差异化**：在企业 RAG 赛道中，关系推理能力可作为演示亮点。
- **架构可扩展**：为后续 Agent 多跳推理、智能体工作流打下基础。

### 成本回收条件

建议通过以下指标判断是否值得继续投入：

| 指标 | 目标值（试点后评估） |
|---|---|
| 关系型问题占比 | ≥ 15% 的用户查询涉及跨实体关系 |
| GraphRAG 召回准确率 | ≥ 75%（子图包含正确答案所需的关键实体与关系） |
| 用户满意度 | 关系型问题满意度较基线提升 ≥ 20% |
| 平均响应时间 | ≤ 现有 RAG 链路的 2 倍 |

## 试点方案

### Phase 1：概念验证（2 周）

- 选择一个实体关系密集的知识库（建议：企业组织架构/高管履历）。
- 手动标注小规模 gold set（20–50 个关系型问题）。
- 使用轻量级图存储（如 Neo4j 社区版或 NetworkX + 内存索引）验证抽取与检索效果。
- 输出：抽取准确率、检索准确率、响应延迟。

### Phase 2：工程 MVP（4–6 周）

- 设计知识库级 GraphRAG 开关配置。
- 实现自动实体/关系抽取 pipeline（可基于 LLM + 少量规则）。
- 实现查询路由：向量 RAG vs GraphRAG。
- 集成到现有生成链路，统一引用展示。
- 输出：可演示的端到端关系问答能力。

### Phase 3：评估与扩展（2–4 周）

- 在真实用户查询中抽样评估。
- 决定是否扩展至第二、第三个知识库。
- 完善增量更新、监控、fallback 机制。

## 范围边界

### 在范围内（MVP）

- 知识库级别的 GraphRAG 启用开关；
- 一个试点知识库的端到端 GraphRAG 链路；
- 基于 LLM 的实体/关系抽取（允许人工审核或后处理）；
- 查询路由与结果回退机制；
- 与现有引用展示框架的集成。

### 不在范围内（MVP）

- 全自动、零监督的通用图谱构建；
- 多语言/跨知识库的统一大图；
- 复杂的多跳推理 Agent；
- 实时图谱更新（MVP 可先接受分钟/小时级延迟）。

## 成功标准

| 标准 | 衡量方式 |
|---|---|
| 试点知识库关系型问题回答准确率 ≥ 75% | 人工评估 gold set |
| 关系型答案包含可解释路径 ≥ 80% | 系统自动标注 + 抽样检查 |
| 平均响应延迟 ≤ 向量 RAG 的 2 倍 | 性能测试 |
| 至少一个演示场景可被外部观众理解 | 内部演示/面试展示 |

## 下一步行动

1. **确认试点知识库**：选择一个内容质量高、关系型问题明确的知识库。
2. **跑通技术可行性研究（TR）**：由架构师细化技术方案、存储选型、抽取策略。
3. **收集真实问题样本**：从现有 query 日志中筛选关系型问题，建立评估基准。
4. **启动 Phase 1 PoC**：在小样本上验证抽取与检索效果。
5. **召开决策评审会**：根据 PoC 结果决定是否进入 Phase 2 MVP。
