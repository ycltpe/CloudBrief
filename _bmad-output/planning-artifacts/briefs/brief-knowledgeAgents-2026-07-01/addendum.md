# 附录：下游文档（PRD / 架构）可用的深度素材

> 这些内容不属于产品简报正文（简报只讲"做什么、为谁、为何"），但对后续 PRD 与架构设计有价值，先在此沉淀。

## 技术底座（来自前期 grounding 调研，已验证为主流方案）

Enterprise RAG 的成熟检索链路：

```
用户提问
  → 混合检索：BM25(关键词/精确术语) + 向量语义检索(同义/换句话说)
  → RRF 融合排序（Reciprocal Rank Fusion，把两路结果融合）
  → Reranker 重排（cross-encoder 对候选片段二次精排）
  → LLM 带引用生成（基于 top 证据，逐句标注出处）
  → 诚实拒答（证据不足时不编造）
```

## 待架构阶段决策的技术选型（Mary 将推荐 Python 为主）

- **向量库**：候选 Chroma / FAISS / Qdrant（作品集优先易上手、依赖少）。
- **关键词检索**：BM25（rank-bm25 库）或 Elasticsearch（后者偏重）。
- **Embedding 模型**：开源 bge 系列 / 商用 API，二选一。
- **Reranker**：bge-reranker 等 cross-encoder。
- **LLM**：默认使用最新最强的 Claude 模型（如 Opus 4.8 / Sonnet 5），面试可讲清为何选它。
- **编排**：LangChain / LlamaIndex，或轻量自研（作品集里"自研"更能体现理解深度）。
- **后端**：FastAPI；前端：简单 Web UI 或 CLI 演示。

*具体选型在架构阶段（Winston）定稿并说明取舍理由。*

## 竞品参照（来自前期调研，作品集面试可引用）

- **Glean** 是 Enterprise Search/RAG 的标杆产品，但价格高、不支持自托管。
- 微软 **Copilot** 系是另一类竞品，绑定 M365 生态。
- 作品集不与它们竞争，仅作为"证明这个方向有真实市场"的参照。

## 已放弃的方向（避免重走）

- 原 B2B SaaS / 私有化 / 气隙部署 / 强监管行业方向已放弃——那些卖点无法用代码演示，会让作品集"嘴大手小"。本案例聚焦可演示的 RAG 核心。

## RAG 三大难点（面试重点，MVP 必须正面处理）

1. **溯源引用**：答案逐句可追溯到具体文档/工单，支持人员与质检可复核。
2. **诚实拒答**：无证据时明说"不知道"，是 B 端信任底线，也是评测核心指标。
3. **答案时效**：附来源更新时间，缓解"文档更新未同步"导致的过期答案。
