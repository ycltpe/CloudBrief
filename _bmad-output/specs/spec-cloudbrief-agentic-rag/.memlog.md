- (decision) 采用 LangGraph StateGraph 作为 Agentic RAG 编排层：1.0（2025-10）起 API 稳定承诺、调研日核验 1.2.9 活跃；LlamaIndex Workflows 活跃但核心包仍 0.x 落选；自研 DAG 控制流不可审计落选
- (decision) 架构原则：图内确定性路由 + 节点内受限自主；护栏（拒答阈值、权限校验、跳数终止、时效检查）由代码控制的条件边实现，正确性不依赖 LLM 行为
- (decision) 双路径并行（Strangler Fig）：新增 orchestration_mode 设置项（native/langchain/agentic，默认 native），照搬 SettingMeta 注册模式；回滚 = 后台开关秒级切换，无数据迁移
- (decision) 按 Phase 0-4 五阶段渐进落地（依赖+开关 → 等价平移 → CRAG 纠偏 → 工具化多跳 → 可选 checkpointer），每阶段含退出条件，shadow 对照 + RAGAS 回归守门，未过门不进下一阶段
- (decision) SSE 协议零改动：LangGraph stream_mode 映射现有六事件协议（messages→chunk、updates→status、custom→sources、图结束→citations+done），前端不动
- (decision) 工具调用走路线 A：扩展 ModelClient.chat(tools=...)，保住 provider failover/重试/统一日志；不引入 langchain-openai ToolNode 路线（Phase 1-2 不需要 tool calling）
- (decision) 会话消息继续存 MySQL；checkpointer 只存编排状态（thread_id == conversation_id），Phase 4 才引入，先 SQLite 零运维验证，需要分布式再迁 Redis（复用 6381）
- (decision) 编排轨迹落 query_logs.tool_trace（路由决策、工具调用序列、跳数、各跳 max_score、模型/延迟/token），prometheus 指标加 orchestration_mode 标签；LangSmith 仅开发期调试，不进生产关键路径
- (decision) 图 State 复用现有 RetrievalResult Pydantic 契约，无 DTO 转换层；StateGraph 定义集中于新目录 app/orchestration/，编译为模块级单例
- (constraint) JWT 鉴权与 kb 权限校验在图入口之前完成，不下沉为 Agent 可决策节点，避免路由自由度变成越权面
- (constraint) 不新增服务：LangGraph 以库形态嵌入现有 FastAPI 进程，docker compose 清单与 Celery 队列不变
- (constraint) 检索循环 max_hops=2 封顶，第二跳起 top_k 减半；plan/grade 节点低 temperature + 短 max_tokens + JSON 结构化输出（可降级到便宜模型档）
- (constraint) Stage 契约与现有 Stage 测试不动；langchain 生态被带至 1.x 后先验证 LangChainRetrievalStage 再继续后续阶段
- (constraint) SSE 六事件协议（chunk/citations/status/done/error/sources）与前端消费逻辑不变；编排逻辑全部在 Python 后端
- (non_goal) ReAct 全自主循环（LLM 自主决定何时检索、检索什么、何时停）不作为目标架构
- (non_goal) 不引入 LangGraph Platform（托管/自部署 server），不引入生产环境 LangSmith 依赖
- (non_goal) 不采用 create_agent 黑盒抽象，不做多 agent 分工协作（检索 agent + 生成 agent）
- (non_goal) 不做 Self-RAG 式 reflection tokens 微调（思想借用、实现不借用）；不引入 llama-agents（已弃用）
- (non_goal) 前端无改造：不新增页面、不改交互；编排价值全部经后端与可观测数据体现
- (capability) CAP-1: 管理员可在运行期切换编排模式（native/langchain/agentic，默认 native）→ 后台切换即时生效，native 默认下行为与现状一致，config_snapshot 记录于 query_logs
- (capability) CAP-2: agentic 模式下系统以状态图执行与现有流程等价的问答（改写→检索→生成），SSE 协议不变 → 现有 pytest 全绿且同一 RAGAS eval 集上 faithfulness/answer relevancy 与 native 持平（Phase 1 退出条件）
- (capability) CAP-3: 系统评估检索质量并在低分时改写查询重试（max_hops=2），拒答判定保持代码硬分支 → eval 集上低质检索用例的回答率/拒答准确率改善可量化（Phase 2 退出条件）
- (capability) CAP-4: 系统按查询复杂度三档路由（直接检索/多跳分解/图谱增强）并将检索工具化 → eval 集多跳样本通过且每次路由决策与工具调用序列可从 query_logs 回放审计（Phase 3 退出条件）
- (capability) CAP-5: 系统记录每次请求的编排轨迹（tool_trace）并纳入现有指标管道 → tool_trace 覆盖率 100%，prometheus 按 orchestration_mode 标签可对比双路径延迟分布
- (capability) CAP-6: 系统可跨请求持久化编排状态并支持中断恢复（可选，Phase 4）→ checkpointer 启用后跨请求状态可恢复（Phase 4 退出条件）
- (assumption) 每个 agentic 节点 +0.5-2s 延迟为经验量级（置信度中），Phase 1 后以本项目实测校准
- (assumption) Adaptive RAG 论文编号（arXiv 2403.14403）未逐字核验，三档路由设计依据工程需要而非论文细节
- (assumption) eval 集可扩充出足量多跳问题样本，否则 CAP-4 的价值无法被度量
- (question) Phase 4（checkpointer + 中断恢复）的启动触发条件是什么信号？无明确需求信号则不做
- (question) 多跳评测样本的来源与规模如何确定？（eval 集扩充是 CAP-4 度量的前提）
- (note) 成功信号：同一 RAGAS eval 集双路径对照——agentic 的 faithfulness/answer relevancy 不降、低质用例拒答准确率提升、多跳样本通过；P50 总延迟增幅 < 30%（单跳）、TTFB 不回退、平均 token/请求增幅 < 2x（多跳样本除外）；tool_trace 覆盖率 100% 且 hop 分布可见
- (decision) graph 工具沿用 GraphSchemaStore 的 per-kb 开关（enabled/shadow_mode）决定是否向 plan 节点提供图谱档，不新增图访问控制面
- (decision) plan/grade 节点 prompt 按 QueryRewriteStage 同模式资产化管理（独立 stage 文件 + 模板常量），随 query_logs 采样复盘迭代
- (decision) 同步 Stage 在图节点中继续 asyncio.to_thread 包裹，多跳子问题可并行检索；图编译单例无共享可变状态，uvicorn 多 worker 安全
