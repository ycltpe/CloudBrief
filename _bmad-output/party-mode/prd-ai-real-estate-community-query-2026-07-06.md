# PRD：AI 房产助手 —— 小区名称查询功能

**版本：** v1.0  
**日期：** 2026-07-06  
**状态：** 草案（待评审）  
**负责：** BMad Party Mode 圆桌产出  
**试点城市：** 上海（MVP 限定）

---

## 1. 背景与目标

### 1.1 背景
用户在 AI 房产助手中输入一个小区名称（如“和合苑”），通常处于看房决策的早期阶段：他可能在中介嘴里听到这个名字，或在某篇文章里瞥见，想快速建立对这个小区的直觉。此时的核心需求不是立即看房，而是**在 60 秒内获得可信、结构化、可追问的小区画像**。

### 1.2 产品目标
- **目标 1（主）：** 当用户输入小区名时，系统能在 2 秒内给出针对该小区的、结构化的、可解释的综合画像。
- **目标 2：** 当存在同名或近似名小区时，系统优先消歧，不强行给出错误答案。
- **目标 3：** 所有关键数据必须标注来源与更新时间，缺失时明确说明，禁止编造。

### 1.3 成功指标
- 小区名单轮查询的完整画像返回率 ≥ 90%（试点城市内）。
- 消歧场景下用户成功选择并继续对话的比例 ≥ 70%。
- 用户对“数据来源/更新时间”可见性感知度 ≥ 80%（通过满意度抽样）。

## 2. 用户故事与 JTBD

### 2.1 核心用户故事
> 作为一个正在看房的人，  
> 我想在输入小区名称后快速了解这个小区的综合情况，  
> 以便决定要不要继续看这里的房源。

### 2.2 Jobs-to-be-Done

| JTBD | 用户目标 | 产品责任 |
|------|----------|----------|
| JTBD-1 建立直觉 | 知道这个小区大概“是什么档次、在哪、多少钱” | 首屏给出位置、均价、核心标签 |
| JTBD-2 核实关键信息 | 确认学区、交通、生活配套 | 展示关键配套，标注来源与更新时间 |
| JTBD-3 判断时机 | 判断现在买/卖/租是否合适 | 展示近期趋势、挂牌量、成交周期等 |
| JTBD-4 消除不确定性 | 了解最坏情况（利率涨、房价跌） | 提供可追问的压力测试入口（V2） |

### 2.3 典型场景

- **场景 A（唯一匹配）：** 用户输入“和合苑”，系统识别为上海浦东的和合苑，直接返回小区画像卡片。
- **场景 B（多候选）：** 用户输入“和合苑”，系统发现上海有 3 个同名/近似小区，展示候选列表让用户选择。
- **场景 C（无匹配）：** 用户输入的小区名在数据库中不存在，系统诚实告知，并给出下一步建议（如换个名称、补充城市）。
- **场景 D（错字/别名）：** 用户输入“合和苑”，系统通过别名/拼音匹配到“和合苑”，并提示“您是不是想找：和合苑”。

## 3. 功能范围

### 3.1 MVP（Phase 0 + Phase 1）
- 小区名称输入解析与实体识别。
- 基于城市/用户上下文的小区消歧。
- 单匹配时返回小区画像卡片：位置、均价、户型、核心标签、学区、交通、数据来源/更新时间。
- 多匹配时返回候选列表，支持一键选择。
- 缺失数据时显示“暂无数据”并说明原因。
- 支持用户追问：“最近的成交价是多少？”“学区稳定吗？”（仅返回已接入数据，不推断）。

### 3.2 明确不做（MVP 外）
- 房价预测 / 投资建议 / 升值承诺。
- 自动联系中介或预约看房。
- 全国所有城市同时上线（MVP 仅限上海）。
- VR 看房、户型图生成、贷款计算器等重交互功能。
- 替代法律、税务、贷款专业建议。

### 3.3 V2 规划
- 动态成交/挂牌数据接入。
- 租金回报率、成交周期、涨跌趋势。
- 购房压力测试（利率上浮、失业、房价跌 20% 场景）。
- 多城市扩展。

## 4. 信息架构：小区画像卡片

当系统对“和合苑”达成唯一高置信匹配时，返回以下结构（首屏仅展示 L1，L2 可展开）：

### 4.1 首屏（L1）—— 一眼看完

| 字段 | 示例 | 说明 |
|------|------|------|
| 小区名 | 和合苑 | 主名 + 常用别名 |
| 核心标签 | 刚需上车盘 / 地铁 500m / 学区老破小 | 由算法生成，但必须基于可验证数据 |
| 位置 | 上海市浦东新区 XX 街道 | 精确到街道，可跳转地图 |
| 参考均价 | 6.8 万/㎡ | 标注“参考”与数据来源 |
| 主要户型 | 55–89㎡ 两房、90–120㎡ 三房 | 基于挂牌样本 |
| 数据时间 | 2026-06 更新 | 统一标注 |

### 4.2 展开层（L2）—— 进一步了解

- **学区：** 对口学校（小学/初中）、学区政策风险说明、数据来源。
- **交通：** 最近地铁站、距离、公交线路概览。
- **生活配套：** 医院、商场、公园（取最近 1km 内 POI）。
- **小区基础：** 建成年代、容积率、绿化率、物业类型、总户数。
- **市场信号（V2）：** 近 3 月成交、挂牌量、成交周期、环比涨跌。

### 4.3 数据来源与透明度
- 每一项数据后必须带 `[来源：XX，更新：YYYY-MM]` 或 `[暂无数据]`。
- 用户可点击“为什么这样总结”查看生成逻辑与数据限制。

## 5. 交互流程

```
用户输入："和合苑"
        ↓
[意图识别] → 识别为"小区查询"
        ↓
[实体解析] → 小区名 = "和合苑"，候选城市 = 上海（默认/用户上下文）
        ↓
[消歧判断]
    ├─ 唯一高置信匹配 → 直接返回小区画像卡片
    ├─ 多个候选 → 返回候选列表，用户点击后返回卡片
    ├─ 无匹配 → 返回"未找到该小区" + 建议
    └─ 错字/别名 → 提示"您是不是想找：和合苑"
        ↓
用户可追问："最近的成交价？" / "学区对口哪所？" / "这里适合刚需吗？"
        ↓
系统仅基于已接入数据回答，无法回答时说明限制
```

## 6. 消歧规则

### 6.1 消歧触发条件
当小区名在数据库中对应多个记录，或置信度低于阈值时，触发消歧。

### 6.2 消歧排序逻辑
1. **用户当前城市/IP 城市** 优先。
2. **用户历史查询城市** 其次。
3. **同名小区中挂牌/成交样本更多** 的优先。
4. **距离用户当前位置更近** 的优先（如已授权定位）。

### 6.3 消歧展示形式
```
“和合苑”在上海市有 3 个候选，您指的是哪一个？

• 和合苑 · 浦东新区 · 近地铁 2 号线
• 和合苑 · 闵行区 · 2010 年建成
• 和合苑 · 松江区 · 动迁安置房

或者，输入“浦东和合苑”让我更精准地找。
```

### 6.4 别名与错字处理
- 维护小区别名表（如“和合苑”又名“和合家园一期”）。
- 支持拼音/同音字/形近字模糊匹配，匹配成功后需提示用户确认。
- 置信度低于 0.7 时，不自动纠偏，改为询问。

## 7. 数据需求

### 7.1 主数据
- 小区基础表：名称、别名、城市、区、街道、经纬度、建成年代、物业类型、总户数、容积率、绿化率。
- 学校对口表：小区 → 对口小学/初中、政策年份、备注。
- POI 表：地铁站、公交站、医院、商场、公园及距离。

### 7.2 市场数据（MVP 可用静态快照，V2 实时接入）
- 挂牌均价、户型分布。
- 近 3 月成交样本（V2）。
- 挂牌量、成交周期（V2）。

### 7.3 数据质量要求
- 所有数据项必须可追溯到来源系统或人工录入记录。
- 数据更新频率在卡片上明示。
- 缺失字段不得用模型生成内容填补。

## 8. 非功能需求

| 维度 | 要求 |
|------|------|
| 响应时间 | 小区名单轮查询 P95 ≤ 2s（不含用户网络延迟） |
| 可用性 | 数据服务降级时，至少返回小区基础信息，不整体失败 |
| 可解释性 | 所有 AI 总结必须可追溯到数据字段与规则 |
| 合规性 | 不承诺房价走势、不提供投资建议、关键信息标注来源 |
| 可扩展性 | 消歧与卡片渲染模块独立，便于接入新城市 |
| 可测试性 | 每个 AC 都对应可自动或手动验证的用例 |

## 10. 风险与法律声明

### 10.1 产品原则
- AI 助手提供的是**信息整理与参考**，不是房产投资建议。
- 学区、成交、政策等信息可能随时间变化，用户应以官方渠道为准。
- 任何价格数据均为“参考均价”，不构成交易依据。

### 10.2 主要风险
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 消歧错误导致推荐错小区 | 高：用户可能基于错误信息决策 | 多候选时强制用户确认；单匹配低置信时仍提示 |
| 学区信息错误 | 高：影响购房重大决策 | 明确标注数据来源与政策年份；提示“以教育局公示为准” |
| 模型编造数据 | 高：信任崩塌、法律风险 | 缺失字段显示“暂无数据”；关键字段不走 LLM 生成 |
| 数据更新滞后 | 中：价格/政策与市场脱节 | 明示更新时间；V2 接入实时数据源 |
| 用户过度依赖 AI 结论 | 中：忽略专业建议 | 在关键节点提示咨询中介、律师、银行 |

## 11. 实现路径

### Phase 0：基础与消歧（2 周）
- 搭建小区主数据表（上海试点）。
- 实现小区名实体识别与别名/错字匹配。
- 实现消歧逻辑与候选列表渲染。
- 接入基础位置与建成年代数据。

### Phase 1：小区画像卡片（2 周）
- 定义卡片信息架构与 UI 组件。
- 接入学区、交通、生活配套数据。
- 实现均价、户型分布统计（基于挂牌快照）。
- 实现数据来源/更新时间标注。
- 完成追问接口框架（仅返回已有数据）。

### Phase 2：动态市场数据（V2）
- 接入成交、挂牌、趋势数据。
- 实现涨跌、成交周期、挂牌量等市场信号。

### Phase 3：个性化与扩展（V2）
- 购房压力测试。
- 多城市扩展。
- 用户偏好学习与主动推荐。

## 12. 失败模式与 Fallback

| 场景 | 系统行为 |
|------|----------|
| 数据库中无该小区 | 返回：“未找到‘XX’相关小区。你可以尝试：补充城市名（如‘浦东和合苑’）、检查小区名称、或告诉我你在看的房源链接。” |
| 多候选但用户不选择 | 保留候选列表，用户可继续输入更精确名称；不默认选择第一个 |
| 某字段数据缺失 | 显示“暂无数据”，不影响其他字段展示 |
| 数据服务降级 | 返回小区基础信息 + 提示“部分数据暂时不可用” |
| 模型置信度低 | 不生成总结，改由规则模板返回；必要时转人工 |
| 用户追问超出现有数据 | 诚实回答“这个问题我暂时没有足够数据回答”，并建议可获取信息的渠道 |

## 13. 附录

### 13.1 术语表
- **小区画像卡片：** 针对单一小区的总结性信息展示单元。
- **消歧：** 当输入对应多个可能实体时，系统引导用户明确目标的过程。
- **参考均价：** 基于当前可获取挂牌样本的价格区间，非官方评估价或交易指导价。

### 13.2 待决策事项

- [ ] 学区数据更新频率要求（季度/半年）？
