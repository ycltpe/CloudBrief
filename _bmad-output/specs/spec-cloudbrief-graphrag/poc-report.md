# CloudBrief GraphRAG Phase 1 PoC 评估报告

> 评估日期：2026-07-09  
> 评估目标：验证 GraphRAG 在关系型问题上的补充价值，以及 Neo4j + LLM/启发式抽取 + 生成阶段增强管线的可行性。

## 1. 执行摘要

本次 PoC 在 3 个典型场景样本知识库（组织架构、高管履历、供应链）上完成端到端 GraphRAG 链路验证：

- **基线（仅向量 RAG chunks）准确率：93.3%（14/15）**
- **GraphRAG（chunks + 子图上下文）准确率：86.7%（13/15）**
- **GraphRAG 额外耗时：约 105.6%（平均 9.4s → 19.3s）**

PoC 证明：
1. Neo4j 图存储、kb_id 隔离、参数化 Cypher 查询可稳定运行。
2. 子图上下文能正确注入生成阶段 prompt，并在部分关系型问题（如跨文档履历、供应商-客户关系）上提供结构化关系路径。
3. 当前主要瓶颈是 **实体链接 LLM 调用** 与 **抽取阶段稳定性/成本**。

> 注：由于 DashScope qwen3.7-plus 在本次评估后期出现 403/超时，PoC 采用了一份**人工整理的抽取结果缓存**来跑通评估管线。LLM 自动抽取的准确率、召回率仍需在 API 稳定后单独评估。

## 2. 测试设计

### 2.1 样本知识库

| 场景 | 文档 | 核心内容 |
|---|---|---|
| 组织架构 | `sample_org.md` | 云景科技管理层、部门设置、汇报关系 |
| 高管履历 | `sample_executives.md` | 5 位高管的教育与任职经历 |
| 供应链 | `sample_supply_chain.md` | 核心供应商、客户、渠道与合作关系 |

共 3 篇文档，切分为 4 个 chunk。

### 2.2 Gold Set

`gold_set.json` 共 15 道关系型问题，覆盖：

- 汇报关系（如“张伟的直接下属有哪些？”）
- 跨文档共同履历（如“王建国和张伟是否在同一家公司任过职？”）
- 供应商/客户关系（如“东方国信代理哪些公司的产品？”）

### 2.3 评估方式

- **基线**：使用关键词匹配的 top-k chunks 作为上下文，调用 LLM 回答。
- **GraphRAG**：在基线 chunks 上额外注入 1-2 跳子图上下文，再调用 LLM 回答。
- **评判**：使用 LLM judge 对每道题给出 correct / partial / incorrect  verdict，并辅以关键词兜底。

### 3.3 关键观察

1. **GraphRAG 并未在所有问题上超越基线**：由于样本文档较短、chunks 本身已包含答案，基线表现已经很高。GraphRAG 主要在跨 chunk 关系拼接场景提供结构化证据。
2. **耗时偏高**：GraphRAG 每条问题增加一次实体链接 LLM 调用 + 一次 Neo4j 子图查询，平均耗时翻倍。当前实现未做实体缓存/预计算。
3. **Q12 失败原因**：问题要求“云景科技的供应商”被“长江制造使用”，需要两跳推理。当前子图查询返回的是分别与“云景科技”和“长江制造”相关的子图，未正确拼接两条关系路径。

## 4. 抽取质量

由于 API 不稳定，本次 PoC 使用**人工整理**的 57 实体 / 60 关系作为图数据。若按 LLM 抽取评估：

- 第一次 LLM 抽取成功时（qwen3.7-plus）产出 **57 实体 / 66 关系**，耗时约 53s。
- 后续 LLM 抽取出现 403/120s 超时，说明当前 DashScope 调用在长文本抽取任务上不够稳定。
- 抽取准确率、召回率需在 API 稳定后使用 gold set 中的 key_entities / key_relations 进行人工抽样评估。

## 6. 风险与优化建议

### 6.1 主要风险

1. **抽取稳定性**：LLM 抽取长文本成本高、耗时长、易受 API 波动影响。
2. **实体链接开销**：每问一次 LLM 实体链接是主要耗时来源。
3. **多跳推理能力有限**：当前 1-2 跳子图查询无法处理复杂的“云景科技供应商 → 客户使用”类问题。

### 6.2 Phase 2 优化方向

1. **混合抽取策略**：LLM 负责 schema 推荐与复杂关系，规则/spaCy 负责高频实体与关系，降低成本。
2. **实体链接缓存**：对问题中高频实体做本地缓存，减少重复 LLM 调用。
3. **子图查询优化**：支持显式多跳模板（如“某实体的供应商的客户”），提升 Q12 类问题表现。
4. **Shadow mode**：在 GenerationPipeline 中同时调用 GraphRAG 但不影响最终答案，持续收集差异数据。
5. **SSE 进度事件**：在 `rebuild_graph_task` / `index_file_graph_task` 中发布 `graph_extraction` / `graph_building` / `graph_indexing_complete` 事件。

## 7. 结论

GraphRAG 技术在 CloudBrief 场景下**技术上可行**，能够为关系型问题提供结构化关系路径。但在当前实现中，

- **准确率提升空间有限**（基线已很高的小型样本）；
- **耗时开销较大**（翻倍），主要来自实体链接 LLM 调用；
- **抽取稳定性**是最大工程风险。

建议 Phase 2 以**可选开关 + Shadow mode**方式接入主流程，在真实 query 日志上持续评估 ROI，再决定是否全面推广。
