## Story 3.1：第二个知识库试点

**目标：** 将 GraphRAG 扩展到第二个知识库并复用相同评估流程，验证方案在非样本场景下的通用性。

**关联需求：** FR1、FR2、FR3、FR4、FR5、FR8

### Task 3.1.1 选择第二个试点知识库

- [ ] **描述**：在现有 CloudBrief 数据或新上传文档中，选择第二个与 PoC 场景差异明显的知识库（如客服 SOP、产品手册、财务制度等）。
- [ ] **验收标准**：
  - 文档已上传并完成向量索引。
  - 明确记录该知识库的业务场景、文档类型、chunk 数量。
  - 与第一个试点场景不重复。

### Task 3.1.2 为第二个知识库生成并确认 schema

- [ ] **描述**：基于该知识库 chunks 采样调用 schema 自动推荐服务，由管理员确认或微调 entity_types / relation_types。
- [ ] **验收标准**：
  - 自动生成 schema 推荐成功。
  - 管理员完成确认/微调并保存到 `kb_graph_schemas`。
  - schema 覆盖至少 3 类实体与 2 类关系。

### Task 3.1.3 执行第二个知识库的全量图索引构建

- [ ] **描述**：在管理后台为该 KB 启用 GraphRAG 并触发「重建图谱」，观察 Celery + SSE 全链路是否正常。
- [ ] **验收标准**：
  - 构建任务完成且无错误。
  - SSE 事件流包含 `graph_extraction`、`graph_building`、`graph_indexing_complete`。
  - Neo4j 中按 `kb_id` 隔离写入实体/关系。
  - 构建成功率符合 NFR8（≥ 95%）。

### Task 3.1.4 准备第二个知识库的 gold set

- [ ] **描述**：围绕该知识库的关系型信息，人工编写 20–30 条问题及标准答案，标注是否依赖关系路径。
- [ ] **验收标准**：
  - gold set 覆盖跨实体关系、层级关系、时间线关系等至少 2 种关系类型。
  - 每条问题标注期望引用的 chunk / 子图路径。

### Task 3.1.5 运行第二个知识库的对比评估

- [ ] **描述**：在 gold set 上分别运行「纯向量 RAG」与「向量 RAG + GraphRAG」，记录准确率、子图召回率、耗时。
- [ ] **验收标准**：
  - 关系型问题准确率 ≥ 75%。
  - 子图召回率 ≥ 75%。
  - GraphRAG 额外耗时 ≤ 向量 RAG 总耗时的 30%（NFR7）。

### Task 3.1.6 输出第二个知识库试点评估报告

- [ ] **描述**：汇总 Task 3.1.5 的数据，与第一个试点/PoC 结果对比，分析场景差异对效果的影响。
- [ ] **验收标准**：
  - 报告包含准确率、召回率、耗时、错误样例分析。
  - 明确指出该场景是否适合全面推广 GraphRAG。

## Story 3.2：图谱增量更新

**目标：** 单文件上传后只更新该文件相关的图谱子集，避免每次全量重建。

**关联需求：** FR3、FR4、FR6

### Task 3.2.1 设计增量更新策略

- [ ] **描述**：确定按 `doc_id` / `chunk_id` 标记实体/关系的图模型方案，设计删除旧数据、写入新数据的流程。
- [ ] **验收标准**：
  - 产出设计文档，明确节点/关系属性中如何记录 `doc_id` 与 `chunk_id`。
  - 明确增量更新与全量重建的触发条件。
  - 评审通过（至少 1 名后端开发 + 1 名架构师）。

### Task 3.2.2 扩展 GraphStore 支持按 doc_id 删除

- [ ] **描述**：在 `app/stores/graph_store.py` 中新增 `delete_entities_by_doc(kb_id, doc_id)` / `delete_relations_by_doc(kb_id, doc_id)` 等方法。
- [ ] **验收标准**：
  - 所有 Cypher 查询参数化，强制过滤 `kb_id`（NFR1、NFR2）。
  - 删除后该 `doc_id` 对应的实体/关系不再出现在子图查询中。
  - 不影响其他 `doc_id` 的数据。
  - 单元测试覆盖。

### Task 3.2.3 修改图索引任务支持增量模式

- [ ] **描述**：扩展 `app/tasks/graph_indexing.py`，使 `index_file_graph_task` 支持增量模式：先删除旧 doc 图数据，再抽取并写入新数据。
- [ ] **验收标准**：
  - 新增 `incremental` 参数或单独任务函数。
  - 仅处理传入文件对应的 chunks，不重建整个 KB 的图。
  - 幂等：同一文件重复触发增量更新结果一致。

### Task 3.2.4 在单文件向量索引完成后自动触发增量图更新

- [ ] **描述**：在 `index_file_task` 或 `IndexService` 钩子中，当 KB 启用 GraphRAG 且单文件向量索引成功后，自动提交增量图索引任务。
- [ ] **验收标准**：
  - 未启用 GraphRAG 的 KB 不触发。
  - 触发逻辑与全量重建互斥（避免并发冲突）。
  - 失败时记录日志，不影响向量索引结果。

### Task 3.2.5 发布增量图更新的 SSE 事件

- [ ] **描述**：确保增量图更新任务在事件流中发布 `graph_extraction`、`graph_building`、`graph_indexing_complete` 阶段事件。
- [ ] **验收标准**：
  - 前端 `useTaskStream` 能正确识别并展示增量图更新阶段。
  - 事件进度从 0 到 1 连续更新。

### Task 3.2.6 验证增量更新性能

- [ ] **描述**：在第二个试点知识库上对比「增量更新」与「全量重建」耗时，验证增量更新时间 ≤ 全量重建时间的 20%。
- [ ] **验收标准**：
  - 至少完成 3 次增量上传/更新测试。
  - 记录每次全量重建耗时、增量更新耗时、chunk 数量。
  - 输出性能对比表格。

## Story 3.3：性能与成本监控

**目标：** 监控 Neo4j 查询性能、抽取质量与图谱新鲜度，及时发现并优化瓶颈。

**关联需求：** NFR7、NFR8

### Task 3.3.1 增加 Neo4j 查询耗时监控

- [ ] **描述**：在 `GraphStore` 的关键查询方法（`get_subgraph_context`、`upsert_entities`、`upsert_relations`、删除方法）中埋点，记录查询耗时。
- [ ] **验收标准**：
  - 每次调用记录耗时日志（INFO/DEBUG 级别）。
  - 提供按 `kb_id`、操作类型聚合的平均/最大/最小耗时接口或脚本。
  - 异常慢查询（如超过阈值）单独 WARN 告警。

### Task 3.3.2 增加抽取质量监控

- [ ] **描述**：基于 gold set 或抽样评估，定期计算实体/关系抽取准确率，形成趋势数据。
- [ ] **验收标准**：
  - 提供可周期性运行的评估脚本（可复用 Task 1.11/3.1.5 逻辑）。
  - 输出准确率趋势（至少包含最近 5 次构建结果）。
  - 准确率低于阈值时 WARN 告警。

### Task 3.3.3 增加图谱新鲜度监控

- [ ] **描述**：记录每个 KB 图索引最后更新时间，提供查询接口。
- [ ] **验收标准**：
  - `kb_graph_build_tasks` 或新表记录每次构建完成时间。
  - 管理后台可查看「最后图索引更新时间」。
  - 超过 freshness 阈值（如 7 天未更新）WARN 告警。

### Task 3.3.4 在 admin dashboard 增加 GraphRAG 监控卡片

- [ ] **描述**：在 `app/admin/dashboard` 页面新增 GraphRAG 监控卡片，展示平均查询耗时、抽取准确率、图谱新鲜度。
- [ ] **验收标准**：
  - UI 同时支持明亮/暗黑模式。
  - 数据从后端监控接口读取。
  - 空状态/加载状态处理完善。

### Task 3.3.5 配置关键指标异常日志告警

- [ ] **描述**：定义并配置关键指标异常时的日志告警规则（无需引入新告警系统，先通过结构化日志 + 可选 Sentry/自定义 webhook）。
- [ ] **验收标准**：
  - 明确告警阈值：Neo4j 查询 P99 > X ms、抽取准确率 < Y%、图谱 freshness > Z 天。
  - 阈值可配置（写入 `system_settings` 或 `.env`）。
  - 触发阈值时输出结构化 WARN/ERROR 日志。

## Story 3.4：推广决策报告

**目标：** 基于 gold set 与真实 query 日志输出推广决策报告，决定是否全面引入 GraphRAG。

**关联需求：** 无新增功能需求，属于项目决策交付物。

### Task 3.4.1 收集 Phase 1/2/3 评估数据

- [ ] **描述**：汇总 PoC、Phase 2 MVP、第二个知识库试点的所有评估数据。
- [ ] **验收标准**：
  - 数据包括准确率、召回率、耗时、构建成功率、错误样例。
  - 数据按场景/知识库分类整理。

### Task 3.4.2 基于真实 query 日志分析触发率与回退率

- [ ] **描述**：从 Shadow mode 或生产 query 日志中，分析 GraphRAG 实际触发比例、成功注入比例、回退比例。
- [ ] **验收标准**：
  - 统计 GraphRAG 触发率、子图非空率、阶段异常回退率。
  - 识别高频回退原因（Neo4j 不可用、schema 为空、未识别实体等）。

### Task 3.4.3 估算 GraphRAG 运营成本

- [ ] **描述**：基于 LLM 抽取调用量、Neo4j 资源占用、存储增长，估算单位查询/单位文档的增量成本。
- [ ] **验收标准**：
  - 成本模型包含：LLM token 费用、Neo4j 计算/存储费用、Celery Worker 资源费用。
  - 给出不同规模知识库（1k/10k/100k chunks）的成本估算。

### Task 3.4.4 编写推广决策报告

- [ ] **描述**：整合准确率提升、耗时开销、成本估算、风险与建议，形成最终决策报告。
- [ ] **验收标准**：
  - 报告结构清晰，包含执行摘要、数据汇总、场景分析、成本分析、风险、建议。
  - 报告经过产品/技术/业务三方评审。

### Task 3.4.5 明确给出推广建议

- [ ] **描述**：在报告中明确给出「全面推广 / 继续试点 / 暂停」建议，并说明适用场景与前提条件。
- [ ] **验收标准**：
  - 建议具体、可执行。
  - 列出下一步行动计划（若继续推广，需说明 rollout 节奏；若暂停，需说明重启条件）。

## 风险与缓解

| 风险 | 关联任务 | 影响 | 缓解 |
|---|---|---|---|
| 第二个知识库抽取质量不达预期 | Task 3.1.2、3.1.5 | 高 | 提前做 schema 评审，必要时增加 gold set 人工标注。 |
| 增量更新并发冲突 | Task 3.2.4 | 中 | 使用 Redis 分布式锁保护同一 KB 的图索引任务。 |
| Neo4j 性能瓶颈 | Task 3.3.1 | 中 | 监控 P99 耗时，必要时加索引或限制子图深度/节点数。 |
| 成本高于预期 | Task 3.4.3 | 中 | 提前建立成本模型，必要时引入 spaCy 混合抽取降低 LLM 成本。 |
