# PRD: CloudBrief 支持副驾

## 0. 文档目的

本 PRD 面向下游的架构（Winston）、开发（Amelia）以及作为作品集读者的面试官。它基于已定稿的产品简报（`brief-knowledgeAgents-2026-07-01/brief.md`）展开：简报回答"做什么、为谁、为何"，本 PRD 把它细化为**可开发、可验收的功能需求**。文档以术语表（§3）统一词汇，功能按 Feature 分组、FR 全局编号，推断处以 `[ASSUMPTION]` 内联标注并在 §9 汇总。技术选型不在此文，留给架构文档与 `addendum.md`。

## 1. 愿景

CloudBrief 支持副驾是一个嵌入内部支持工作台的 AI 问答助手。**支持人员**用自然语言提问，系统在四类内部**知识源**（帮助文档、产品更新日志、历史工单、内部 FAQ）上做**混合检索**，由大模型生成一段**带引用**的答案；证据不足时**诚实拒答**而非编造。它支持**多轮对话**，让支持人员能顺着追问。

它不替人做决定——人始终在环。副驾的价值是把"快、准、可溯源、不过期"的答案在几秒内递到支持人员面前，从而缩短响应时间、降低"过期答案"出错率、加速新人上手。

作为作品集，本项目的工程重心是把 RAG 最难的三件事——**溯源引用、诚实拒答、答案时效**——正面做扎实，并用一套可复跑的评测证明效果。

## 2. 目标用户

### 2.1 Jobs To Be Done（用户要完成的任务）

**主要用户：支持人员**
- 当客户抛来一个问题时，**快速找到准确、可信、带出处的答案**，而不用在四个系统间手工翻找。
- **判断答案是否可能过期**（产品迭代快），避免把失效信息发给客户。
- 遇到追问时能**顺着上一个问题接着问**，不用每次重述背景。
- 作为新人，**不靠死记知识分布**也能独立处理工单。

**管理用户：系统管理员**
- 管理可登录系统的用户账号（新增、删除、分配角色）。
- 组织知识库目录结构，上传文件并触发索引重建。
- 查看系统 Dashboard，了解用户活跃度、索引状态与评测趋势。
- 调整系统运行参数（拒答阈值、时效阈值、模型配置等）。

**质检用户：客服主管 / 质检人员**
- 查看 RAGAS 自动评测结果，定位回答质量差的 case。
- 对评测记录进行人工复核与评分，标记是否可采用或需修改。
- 导出人工标注结果，用于后续模型或知识库优化。

### 2.2 非用户（v1 不服务）
- 终端客户（对外自助答疑留给愿景）。
- 复杂的企业级权限体系（v1 只提供最小 RBAC：admin / qa / user）。

### 2.3 关键用户旅程

- **UJ-1. 小林用副驾处理一个棘手工单（多轮）。**
  - **人物与背景**：小林，入职两周的新客服，还没记住知识分布。
  - **进入状态**：已登录支持工作台，手上一张工单——客户问"导出的报表为什么打不开"。
  - **路径**：① 小林在副驾输入框问"报表导出后无法打开怎么办"；② 副驾几秒内给出一段答案，**每个要点后面带可点击的出处**（指向帮助文档某节 / 某张历史工单）；③ 小林追问"那如果是 Excel 格式呢"——副驾**记得上文**，直接就这个格式继续回答；④ 小林点开一条引用核对，确认无误，复制给客户。
  - **高潮**：30 秒内拿到带出处、且他敢直接用的答案，不用问老员工。
  - **收尾**：工单解决；小林对这条知识也更熟了。
  - **边界情况**：若副驾检索不到相关证据，它明说"知识库里没有相关信息"，小林转而走人工升级流程——而不是被一个编造的答案误导。

- **UJ-2. 副驾遇到"可能过期"的知识。**
  - 支持人员问某个功能的操作步骤，副驾找到的最佳证据来自一份**几个月前**的文档。答案照常给出，但**标注来源更新时间**并提示"该来源较旧，请留意产品是否已变更"，支持人员据此多核对一步。

- **UJ-3. 管理员为新团队开通账号并配置知识库。**
  - 管理员登录 /admin 后台，进入"用户管理"新增两名支持人员账号并分配 `user` 角色；再进入"知识库管理"新建"产品更新"目录，上传本季度的 Markdown 文件，点击"整合重建索引"。几分钟后索引重建完成，支持人员即可在聊天助手问到最新内容。

- **UJ-4. 客服主管审计一次低分评测。**
  - 客服主管打开"RAGAS 评测审计"页，发现某条问题 faithfulness 分数较低。点击"查看详情"，看到系统答案、检索到的片段和 RAGAS 判定理由。主管认为答案虽然引用正确但遗漏了关键步骤，给出人工评分 3 分并备注"需补充步骤 4"，标记为"需修改"。

## 3. 术语表
*下游文档与读者必须逐字使用以下术语；PRD 各处（FR、UJ、SM）不得引入同义词。*

- **支持人员（Support Agent）** — 系统的主要用户，公司内部的客服/技术支持，用副驾辅助回答客户问题。
- **副驾（Copilot）** — 本系统「CloudBrief 支持副驾」的简称。
- **知识源（Knowledge Source）** — 四类内部资料：帮助文档、产品更新日志、历史工单、内部 FAQ。副驾只在知识源范围内作答。
- **片段（Chunk）** — 知识源经切分后的最小检索单元，是**引用**指向的对象，带来源标识与更新时间。
- **混合检索（Hybrid Retrieval）** — 同时用关键词检索（BM25）与向量语义检索召回**片段**，再做**融合排序**。
- **融合排序（Fusion）** — 把两路检索结果合并成一个排序列表（如 RRF）。
- **重排（Rerank）** — 用 reranker 对候选**片段**二次精排，提升 Top 结果相关性。
- **引用（Citation）** — 答案中某个论断到具体**片段**的可溯源标注，用户可点开核对。
- **拒答（Refusal）** — 当证据不足以支撑作答时，副驾明确回答"知识库中没有相关信息"，不编造。
- **会话（Conversation）** — 一次多轮问答的上下文序列；副驾在同一会话内记得先前轮次。
- **管理后台（Admin Console）** — 面向管理员/客服主管的 Web 界面，地址为 `/admin`，提供用户管理、知识库管理、系统设置与评测审计功能。
- **用户（User）** — 可登录系统的账号，分为 `admin`（管理员）、`qa`（质检）、`user`（普通支持人员）三种角色。
- **知识库目录（KB Directory）** — 知识源文件在 `data/` 下的物理目录，同时以元数据形式记录在数据库中，用于组织帮助文档、更新日志、工单、FAQ 等文件。
- **评测集（Eval Set）** — 一组带标准答案/标准出处的问题（含可回答与不可回答两类），用于量化副驾效果。

## 4. 功能需求

### 4.1 知识库导入与索引
*把四类知识源切分成可检索的**片段**，建立向量索引与关键词索引，为检索链路提供基础。*

#### FR-1: 支持四种知识源格式导入
[管理员/开发者] 可以导入帮助文档、更新日志、历史工单、内部 FAQ 到系统。MVP 通过离线文件/脚本导入，不支持实时 API 对接。

**可验收条件：**
- 系统能读取 Markdown/JSON/CSV 格式的知识源文件（至少支持一种主要格式）。
- 每种知识源必须保留原始元信息：来源类型、标题、更新时间、原始 URL/ID（如果有）。
- 导入脚本幂等：同一批文件重复导入不会导致片段重复。

**明确不做：**
- 不支持数据库实时同步、Webhook 增量更新、版本控制。

#### FR-2: 文本切分与片段生成
系统把导入的知识源切分为适合检索的**片段**，片段大小兼顾语义完整性与检索精度。

**可验收条件：**
- 帮助文档/更新日志按语义段落切分；工单按工单记录切分；FAQ 按问答对切分。
- 每个片段保留来源元信息（来源类型、标题、更新时间、片段在原文中的位置/ID）。
- 切分策略可配置（如每片段最大 token 数、重叠 token 数），默认策略给出合理结果。

**[ASSUMPTION]**：MVP 用基于 token/段落的简单切分策略，不做基于主题模型的智能切分。

#### FR-3: 建立向量索引与关键词索引（异步任务）
系统为每个片段生成 Embedding 并建立向量索引；同时为关键词检索（BM25）建立倒排索引。索引构建作为**异步任务**暴露给后端，可通过 API 触发，与生成答案请求保持一致的工程形态。

**可验收条件：**
- 向量索引支持语义相似度检索，返回 top-k 候选片段。
- 关键词索引支持 BM25 检索，返回 top-k 候选片段。
- 两类索引可以从同一套片段数据构建，保存到本地文件系统。
- 提供 API 端点触发"导入并重建索引"任务：读取指定目录的知识源文件，切分、生成 Embedding、重建向量索引与 BM25 索引。
- 索引任务为异步执行：提交后立即返回任务 ID，客户端可查询任务状态（pending / running / completed / failed）与失败原因。
- 任务运行期间，已有索引保持可服务；新索引构建完成后原子切换，避免查询中断（或降级为短时不可用，并在任务完成后恢复）。

**明确不做：**
- 不做分布式索引、增量索引更新、多租户隔离。
- 不做复杂调度（如定时重建、增量同步），触发方式为手动/API触发。

---

### 4.2 混合检索与重排
*接收用户查询，召回相关片段，融合排序，重排精筛，输出高质量候选证据。*

#### FR-4: 双路检索召回
对于给定查询，系统同时使用向量语义检索和关键词检索召回候选片段。

**可验收条件：**
- 向量检索路径返回与查询语义相近的片段（即使关键词不完全匹配）。
- 关键词检索路径对精确术语、错误码、产品名更敏感。
- 两路检索结果都包含片段内容与元信息。

#### FR-5: 融合排序
系统将向量检索与关键词检索的结果融合成一个统一的候选列表。

**可验收条件：**
- 使用 RRF（Reciprocal Rank Fusion）或等效方法融合两路结果，RRF 参数 `k=60`。
- 融合后的列表长度可配置，默认取 Top 50。
- 同一文档的多个片段在融合后适当去重或降权。

**[ASSUMPTION]**：默认使用 RRF，参数 `k=60`；架构阶段可与 Winston 确认是否有更优方案。

#### FR-6: Reranker 重排精筛
系统对融合后的候选片段做二次精排，把最相关的片段排到最前。

**可验收条件：**
- 使用 `qwen3-rerank`（或架构阶段确认等效模型）对查询-片段对打分。
- 重排后输出 Top-N 片段（默认 N=5，可配置），作为生成答案的证据。
- 重排分数低于阈值的片段不进入生成阶段（防御性过滤）。

**[ASSUMPTION]**：MVP 使用 `qwen3-rerank` 作为 reranker，具体调用方式（本地部署 / API / HuggingFace 等）由架构阶段确定。

### 4.3 答案生成
*基于重排后的 Top 证据生成答案，强制逐句/逐点可溯源，证据不足时拒答，证据陈旧时提示。*

#### FR-7: 基于证据生成带引用的答案
系统将重排后的 Top-N **片段**作为上下文，调用大模型生成答案。答案中的每个关键论断必须附带**引用**，指向它依据的片段。

**可验收条件：**
- 生成请求必须传入：用户问题、会话上下文（多轮）、Top-N 片段（含内容与元信息）。
- 答案以自然语言返回，结构清晰（如分点、步骤、或一段说明）。
- 每个关键论断后标注**引用 ID**，用户可定位到具体片段。
- 引用格式在 API 响应与 Web 界面中保持一致（如 `[^1]`, `[^2]` 等）。
- 若同一段落混合了多个片段的证据，分别标注对应引用。

**[ASSUMPTION]**：MVP 采用"引用 ID 标注到论断"的粒度，不要求精确到句子内的每个词；具体引用格式由架构阶段确定。

#### FR-8: 诚实拒答
当检索到的证据相关性不足或数量过少时，系统必须**拒答**，而不是基于薄弱证据编造答案。

**可验收条件：**
- 系统根据 reranker 分数、Top-N 片段数量、或专用"相关性门控"判定是否进入拒答分支。
- 拒答时返回固定话术："根据当前知识库，我找不到足够信息回答这个问题。"（文案可配置，语义不变）。
- 拒答响应同时返回**诊断信息**（可选，用于调试/评测）：实际召回片段数、最高 reranker 分数，方便判断拒答是否合理。
- 系统在评测集上的"不可回答问题"中，拒答正确率达到设定基线（如 ≥ 80%）。

**[ASSUMPTION]**：拒答阈值默认由 reranker 分数决定（如 Top 片段最高分数低于某阈值即拒答），具体阈值在评测阶段根据 Eval Set 调优。

#### FR-9: 答案时效提示
当答案所依据的片段中，任一来源的更新时间距离当前时间超过设定阈值时，系统在答案中给出"来源可能过期"的提示。

**可验收条件：**
- 系统读取每个片段的"更新时间"元信息。
- 当答案引用了更新时间早于阈值（如 90 天）的片段时，在答案末尾或对应引用旁附加提示（如"⚠️ 该答案依据的来源较旧，请核对最新产品版本"）。
- 时效阈值可配置，默认 90 天。
- 提示文案不影响答案主体内容，只作为复核提醒。

**明确不做：**
- 不做自动判断"内容是否真实过期"（这需要产品知识图谱或人工标注，超出 MVP）。

### 4.4 多轮对话
*支持人员可以在同一会话中连续追问，副驾记得前文并据此改写检索查询。*

#### FR-10: 会话上下文管理
系统为每次问答创建一个**会话**对象，保存用户与副驾的历史消息（问题与答案）。

**可验收条件：**
- API 支持创建新会话与继续已有会话（通过 `conversation_id`）。
- 历史消息包括：用户提问、副驾答案、引用的片段 ID、拒答标记。
- 会话历史有最大长度限制（如最近 10 轮或最大 token 数），超过时丢弃最早的消息。
- 会话数据持久化到本地存储（文件或 SQLite），服务重启不丢失。

**明确不做：**
- 不做用户认证、会话权限隔离、会话共享。

#### FR-11: 查询改写（Query Rewriting）
在多轮场景中，用户的追问可能依赖前文（如"那 Excel 格式呢？"）。系统在检索前把当前问题与会话上下文合并，生成一个**自包含的检索查询**。

**可验收条件：**
- 对于依赖前文的追问，系统能生成一个语义完整的独立查询（如把"那 Excel 格式呢？"改写为"报表导出为 Excel 格式无法打开怎么办？"）。
- 改写后的查询只用于检索链路，不改变用户看到的原始问题。
- 改写策略可配置：简单策略为拼接历史；进阶策略可调用 LLM 做指代消解/查询扩展。

**[ASSUMPTION]**：MVP 先实现"拼接最近 1–2 轮历史 + 简单指代补全"策略；若效果不佳再升级为 LLM-based 改写。

---

### 4.5 评测与指标
*建立可复跑的评测集与指标脚本，用数据证明 RAG 链路效果。*

#### FR-12: 构建评测集
系统维护一份**评测集**，包含可回答问题、不可回答问题、需要拒答的问题、需要时效提示的问题。

**可验收条件：**
- 评测集至少包含 20–30 条问题（数量可扩展），每条问题标注：期望答案要点、期望引用的片段、是否应拒答、是否涉及时效提醒。
- 评测问题覆盖四类知识源（文档/更新日志/工单/FAQ）。
- 评测集以 JSON/YAML 文件形式维护，便于版本化。

**[ASSUMPTION]**：MVP 评测集为人工构造的合成数据，不追求大规模真实标注。

#### FR-13: 自动评测脚本
系统提供一键运行的评测脚本，对评测集中的每个问题调用副驾，输出指标报告。

**可验收条件：**
- 评测脚本可独立运行，输出 JSON/CSV 报告。
- 核心指标至少包括：
  - **检索命中率（Hit Rate）**：正确答案是否出现在 Top-k 召回片段中。
  - **引用准确率（Citation Accuracy）**：答案引用的片段是否包含对应论断。
  - **拒答正确率（Refusal Accuracy）**：对不可回答问题是否正确拒答，对可回答问题是否未误拒答。
  - **答案时效提示正确率**：对旧来源问题是否正确触发时效提示。
- 报告同时输出每个问题的详细结果，便于定位失败 case。

#### FR-14: 端到端延迟指标
评测脚本记录每个问题从请求到完整响应的耗时。

**可验收条件：**
- 报告包含 P50 / P90 / P95 延迟。
- 在本地运行环境下，常见问题的端到端延迟目标 **≤ 30 秒**（作品集演示环境）。

### 4.6 Web 界面
*基于 `@llamaindex/ui` 构建一个优美的聊天界面，支持提问、多轮追问、引用展示、拒答与重建索引。*

#### FR-15: 聊天主界面
界面提供一个聊天窗口，展示历史消息、用户输入框、发送按钮。

**可验收条件：**
- 使用 `@llamaindex/ui` 的聊天组件（或等效 React 组件）实现界面骨架。
- 界面风格简洁专业，适配桌面端浏览器。
- 用户输入后显示 loading 状态，服务端返回后展示完整答案。

#### FR-16: 引用展示
答案中的**引用**在界面上可交互：用户点击引用标记后，能看到对应片段的原文摘要、来源标题、更新时间。

**可验收条件：**
- 答案里的 `[^1]`、`[^2]` 等标记渲染为可点击元素。
- 点击后弹出/展开片段原文摘要与元信息（来源类型、标题、更新时间）。
- 引用的编号与后端返回的片段 ID 保持一致。

#### FR-17: 多轮追问交互
界面支持在同一聊天会话中连续提问，用户无需手动管理 `conversation_id`。

**可验收条件：**
- 前端自动维护当前会话 ID。
- 每条新消息都发送到同一会话，后端根据会话历史做查询改写。
- 界面区分用户消息与副驾消息。

#### FR-18: 拒答与时效提示展示
当后端返回拒答或时效提醒时，界面按相应样式展示。

**可验收条件：**
- 拒答时显示固定文案（如"根据当前知识库，我找不到足够信息回答这个问题。"），不渲染引用。
- 时效提醒以明显但非打扰的方式展示（如黄色警告条或引用旁的小图标）。

#### FR-19: 重建索引触发入口
界面提供"重建知识库索引"按钮，点击后调用 FR-3 的异步索引任务 API，并展示任务状态。

**可验收条件：**
- 管理/设置页面或工具栏提供"重建索引"按钮。
- 点击后调用 `/api/index/rebuild`（或等效端点），返回任务 ID。
- 界面轮询任务状态并展示：进行中 / 完成 / 失败（含失败原因）。
- 重建期间已有问答功能仍可正常使用，或给出清晰提示。

**[ASSUMPTION]**：重建索引入口可放在简单管理面板中，不一定要复杂权限控制。

### 4.7 管理后台
*为管理员/客服主管提供用户管理、知识库管理、系统设置与 RAGAS 评测审计能力。*

#### FR-20: 用户注册与登录
系统提供用户注册、登录、登出能力，登录成功后返回令牌用于后续请求鉴权。

**可验收条件：**
- 提供 `POST /auth/register` 接口：用户名、密码必填，可选角色（默认 `user`）。
- 提供 `POST /auth/login` 接口：校验用户名密码，返回 JWT access token。
- 提供 `POST /auth/logout` 接口：前端清除 token，后端 token 黑名单可选（MVP 可仅前端清除）。
- 提供 `GET /auth/me` 接口：返回当前用户 ID、用户名、角色。
- 用户名唯一，密码使用 bcrypt 等安全方式哈希存储，禁止明文保存。
- 注册和登录接口对未认证用户开放；其余 admin/qa 专属接口需要鉴权与角色校验。

**明确不做：**
- 不支持 OAuth、SSO、多因素认证。
- 不支持密码找回/重置邮件（需 admin 手动重置或删除重建账号）。

#### FR-21: Dashboard 系统概览
管理后台首页展示系统运行关键指标，帮助管理员快速了解当前状态。

**可验收条件：**
- 展示用户总数、今日新增会话数、最近 7 天会话趋势（可选）。
- 展示当前索引状态：active collection 名称、BM25 索引路径、最近一次重建任务状态。
- 展示最近 5 条 RAGAS 评测记录的平均分数（faithfulness、answer_relevance 等）。
- 展示最近 5 条重建任务概览。
- 后端提供 `GET /admin/dashboard` 聚合接口。

#### FR-22: 用户管理
管理员可以在后台查看用户列表、新增用户、删除用户。

**可验收条件：**
- 用户列表页展示：ID、用户名、角色、创建时间、最后登录时间（可选）。
- 支持按用户名搜索、按角色筛选、分页。
- 管理员可新增用户：填写用户名、初始密码、角色。
- 管理员可删除用户：需二次确认；删除后该用户无法登录；不能删除最后一个 admin。
- 后端暴露 `GET /admin/users`、`POST /admin/users`、`DELETE /admin/users/{id}`。
- 仅 `admin` 角色可调用用户管理接口。

**明确不做：**
- 不支持用户自助修改密码/资料（MVP 由 admin 处理）。
- 不支持用户组、部门隔离。

#### FR-23: 系统设置
管理员可以查看和修改系统运行参数。

**可验收条件：**
- 提供设置项：拒答阈值（`refusal_threshold`）、时效阈值天数（`stale_threshold_days`）、最大历史轮数（`max_history_rounds`）、请求超时（`request_timeout`）。
- 设置可持久化到 `system_settings` 表；应用启动时从数据库加载，覆盖 `.env` 默认值。
- 后端暴露 `GET /admin/settings` 和 `PUT /admin/settings`。
- 修改后立即生效（查询服务读取内存中的配置，或定时刷新）。
- 仅 `admin` 角色可修改设置。

**[ASSUMPTION]**：MVP 中只暴露对运行行为影响明确且安全的配置项，不涉及模型密钥、数据库连接等敏感配置。

#### FR-24: 聊天助手入口
管理后台左侧菜单提供"聊天助手"入口，点击后在右侧加载聊天组件。

**可验收条件：**
- 复用现有 `Chat` 组件，保持与首页一致的问答体验。
- 前端路由为 `/admin/chat`。
- 已登录用户可直接使用；前端自动在请求中携带 token。

#### FR-25: 知识库目录管理
管理员可以在后台管理知识库的目录结构。

**可验收条件：**
- 左侧展示目录树：根目录为 `data/`，支持展开/收起。
- 支持新建目录：指定父目录和目录名；在 `data/` 下创建物理目录，并在 `kb_directories` 表中记录元数据。
- 支持删除目录：仅允许删除空目录；删除前二次确认。
- 后端暴露 `GET /admin/kb/directories`、`POST /admin/kb/directories`、`DELETE /admin/kb/directories/{id}`。
- 仅 `admin` 角色可管理目录。

#### FR-26: 知识库文件管理
管理员可以在指定目录下上传、删除文件，并触发索引重建。

**可验收条件：**
- 文件列表展示：文件名、大小、来源类型、更新时间、操作入口。
- 支持在指定目录上传文件：支持 Markdown/JSON/CSV 格式（与 FR-1 一致）。
- 上传的文件保存到 `data/<dir>/` 下对应物理路径。
- 上传后更新 `kb_files` 表元数据（文件名、目录 ID、来源类型、文件路径、更新时间）。
- 支持删除文件：删除物理文件和元数据记录，二次确认。
- 提供"整合重建索引"按钮：调用现有 `POST /index/rebuild`，基于当前 `data/` 下所有文件重建索引。
- 重建完成后提示"索引已更新"。
- 后端暴露 `GET /admin/kb/directories/{id}/files`、`POST /admin/kb/files`、`DELETE /admin/kb/files/{id}`。
- 仅 `admin` 角色可管理文件。

**[ASSUMPTION]**：删除文件/目录后不自动触发索引重建，而是提示"索引已过期，请手动重建"，避免误删导致长时间任务。

#### FR-27: RAGAS 评测审计
客服主管/管理员可以在后台查看评测记录列表与详情，并进行人工复核。

**可验收条件：**
- 评测列表页展示：问题、答案类型、各 RAGAS 指标分数、人工评分、创建时间。
- 支持按分数范围、是否有人工评分筛选，支持分页。
- 详情页展示：问题（Question）、标准答案（Ground Truth）、系统答案（Answer）、检索到的片段（Contexts）、RAGAS 各指标分数与 reasoning（若可用）。
- 详情页提供人工评分区：1–5 分、备注、是否采用、是否修改。
- 支持导出人工标注结果（CSV/JSON）。
- 后端暴露 `GET /admin/eval/results`、`GET /admin/eval/results/{id}`、`POST /admin/eval/results/{id}/feedback`、`GET /admin/eval/export`。
- `admin` 和 `qa` 角色可查看与复核；`user` 不可访问。

## 5. 非目标（Explicit Non-Goals）

以下能力明确不在本 PRD / MVP 范围内，防止范围蔓延：

- **对外客户自助机器人**：v1 只服务内部支持人员，终端客户问答是愿景。
- **复杂权限与多租户**：v1 只提供最小 RBAC（admin/qa/user），不做用户组、部门数据隔离、细粒度 ACL。
- **私有化部署 / 气隙部署**：本案例聚焦可演示的 RAG 核心，部署形态为本地/单节点演示。
- **增量索引更新与实时同步**：索引重建为全量异步任务，不监听知识源变更。
- **LLM 模型微调**：使用现成模型 API 或开源模型，不对模型做 SFT/RLHF。
- **自动知识库治理**：不自动发现文档缺失、不自动补全 FAQ、不自动标注过期内容。
- **企业级 SLA / 可观测性**：不引入日志平台、监控告警、A/B 测试框架。
- **OAuth / SSO / 多因素认证**：认证采用本地账号 + JWT。

## 6. MVP 范围

### 6.1 In Scope

- 四类知识源的离线导入与切分（FR-1, FR-2）。
- 向量索引 + BM25 关键词索引的异步构建（FR-3）。
- 混合检索、RRF 融合、`qwen3-rerank` 重排（FR-4, FR-5, FR-6）。
- 基于证据的带引用答案生成、诚实拒答、时效提示（FR-7, FR-8, FR-9）。
- 多轮会话与查询改写（FR-10, FR-11）。
- 评测集与自动评测脚本（FR-12, FR-13, FR-14）。
- 基于 `@llamaindex/ui` 的 Web 聊天界面，含引用、多轮、拒答、重建索引入口（FR-15 至 FR-19）。
- 管理后台：基于 JWT 的注册/登录/登出、Dashboard、用户管理、系统设置、知识库目录与文件管理、RAGAS 评测审计（FR-20 至 FR-27）。

### 6.2 Out of Scope for MVP

- 实时多系统对接、增量同步、版本管理。
- 复杂权限、多租户、对外机器人、私有化部署。
- OAuth / SSO / MFA、密码找回邮件、用户自助改密。
- LLM 微调、自动知识治理、企业级可观测性。
- 移动端适配、复杂前端状态管理。

**边界原则**：一切以"能跑、能演示、能把 RAG 核心讲深"为准。

## 7. 成功指标

**Primary**
- **SM-1: 检索命中率** — 评测集中可回答问题的正确答案出现在 Top-5 证据中的比例，目标 ≥ 80%。验证 FR-4, FR-5, FR-6。
- **SM-2: 引用准确率** — 答案中引用与论断一致的比例，目标 ≥ 85%。验证 FR-7。
- **SM-3: 拒答正确率** — 不可回答问题被正确拒答、且可回答问题未被误拒答的比例，目标 ≥ 80%。验证 FR-8。

**Secondary**
- **SM-4: 时效提示正确率** — 旧来源问题被正确提示的比例，目标 ≥ 80%。验证 FR-9。
- **SM-5: 端到端 P90 延迟** — 评测集 P90 响应时间 ≤ 30 秒（本地演示环境）。验证 FR-14。
- **SM-6: 后台认证成功率** — 注册/登录流程可用率 100%，密码必须哈希存储。验证 FR-20。
- **SM-7: 知识库文件上传成功率** — 合法格式文件上传并正确落盘/入库的比例 ≥ 95%。验证 FR-26。
- **SM-8: 评测审计覆盖率** — 评测记录可在后台查看并提交人工反馈，人工反馈接口可用率 100%。验证 FR-27。

**Counter-metrics（不要优化的指标）**
- **SM-C1: 答案长度** — 不要为了"看起来丰富"而生成冗长答案；答案应简洁、可复用。Counterbalances 任何可能鼓励啰嗦的指标。
- **SM-C2: 检索召回数量** — 不要为了提高命中率而无限扩大 Top-k；验证的是"在有限证据内做对"。
