# Product Brief: 文件上传后异步构建索引并实时展示日志

## Executive Summary

在 CloudBrief 支持副驾的管理后台中，管理员上传文件后需要等待“整合重建索引”完成才能让用户检索到最新知识。当前流程存在两个体验断点：

1. **上传与索引之间没有即时反馈**——文件保存到磁盘后，管理员必须手动点击“重建索引”并等待数分钟，期间不知道哪些文件已纳入索引。
2. **索引进度靠轮询刷新**——前端每 1–1.5 秒轮询一次任务状态，日志实时性差、服务器负载高。

本 Brief 定义一个**事件驱动的单文件异步索引**能力：文件上传成功后自动触发 Celery 索引任务，任务执行过程中的步骤与日志通过 SSE 实时推送到前端；同时保留现有的全量重建索引作为兜底与批量操作。该能力直接支撑 Phase 2“增量索引更新”的路线图，并为后续多模态、Agentic RAG 提供可扩展的异步管道基础。

## The Problem

**谁感受到了痛点？**
- **管理员（admin）**：在知识库管理页上传文件后，必须额外点击“整合重建索引”，等待整库重建完成。若只上传 1 个文件，却要重建整个知识库，耗时与成本不匹配。
- **支持人员（user）**：在新文件上传后、重建完成前，无法检索到最新内容，可能给出过期答案。
- **客服主管（qa）**：重建任务失败时，只能在轮询结果中看到最终失败，无法实时定位到哪一步出错。

**现状代价**
- 时间浪费：小更新触发全量重建，平均等待 3–10 分钟（取决于文档量与 Embedding 调用）。
- 认知负担：管理员需要记住“上传后还要点重建”，新用户容易遗漏。
- 可观测性低：轮询间隔导致日志延迟 1 秒以上，快速失败场景排障慢。

## The Solution

构建**“上传即索引”**的异步任务流：

1. **自动触发**：文件上传保存到 `backend/data/kb/` 并写入数据库后，`KbService.upload_file()` 调用 `IndexService.trigger_file_index(file_id)`，将任务投入 Celery 队列。
2. **实时进度**：Worker 执行索引的每一步（解析 → 切分 → Embedding → 写入 Milvus → 更新 BM25 → 原子切换）时，向 Redis Pub/Sub 频道发布事件；FastAPI SSE 端点订阅并推送给前端。
3. **即时可见**：管理后台文件列表旁显示每个文件的索引状态（indexing / indexed / failed），并提供可展开的实时日志面板。
4. **保留重建**：管理员仍可手动触发“整合重建索引”，用于批量更新、模型切换后的全量刷新或增量索引异常时的兜底。

## What Makes This Different

- **渐进增强，不推翻现有架构**：新增 `index_file_task` 与现有 `rebuild_index_task` 共享同一套 Stage（解析/切分/嵌入/写入/切换），避免重复建设。
- **实时体验优于轮询**：项目后端已有 SSE 端点 `/index/tasks/{task_id}/events`，本功能首次让前端真正使用它。
- **事件驱动的可扩展性**：未来要接入 S3 事件、Kafka 流或 Agentic RAG 的触发器，都可以复用同一套任务发布/订阅机制。
- **人在环的可观测性**：每个文件都有独立的任务日志，方便管理员判断“是文件本身有问题，还是索引服务暂时不可用”。

## Who This Serves

**主要用户：管理员（admin）**
- 在 `/admin/kb` 上传、删除、组织知识库文件。
- 期望上传后即时看到“正在索引”与最终结果，无需手动重建。
- 失败时能通过日志快速定位问题文件与错误步骤。

**间接受益：支持人员（user）与客服主管（qa）**
- 支持人员：新知识更快可用，减少给出过期答案的概率。
- 客服主管：RAGAS 评测可覆盖到最新上传的内容，提升评测时效性。

## Success Criteria

| 指标 | 目标 | 说明 |
|---|---|---|
| 上传后任务触发率 | 100% | 文件上传成功后必须成功创建索引任务 |
| SSE 日志延迟 | < 1s | Worker 发布事件到前端收到的时间 |
| 单文件索引成功率 | > 99% | 格式合规、内容可解析的文件应成功索引 |
| 前端轮询次数 | 归零（SSE 路径） | 文件索引状态切换后不再轮询 |
| 幂等性 | 重复上传同一文件不产生重复向量 | 通过 `file_id` + chunk hash 控制 |
| 并发隔离 | 10 并发上传不阻塞全量重建 | 通过队列隔离或分布式锁保证 |

## Scope

### In Scope（v1）

- 文件上传成功后自动触发单文件异步索引任务。
- 复用现有 Stage 实现单文件解析、切分、Embedding、Milvus 写入、BM25 更新、活跃索引切换。
- 通过 SSE 实时推送任务步骤与日志到 `/admin/kb` 页面。
- 文件列表增加 `status` 字段与实时日志入口。
- 保留并兼容现有“整合重建索引”功能。
- 上传接口返回 `task_id`，支持前端订阅。

### Out of Scope（v1，但预留扩展点）

- 增量 chunk hash diff（仅预留字段，v1 可全量重写单文件对应 chunk）。
- 对象存储事件触发（S3/MinIO `ObjectCreated`）。
- 多模态文件（图片/PDF/Excel）解析。
- 任务取消、重试UI、死信队列管理界面。
- Webhook 回调通知外部系统。

## Vision

短期（v1）：上传即索引，管理员在 1 秒内看到任务开始，在数十秒内看到文件变为“已索引”。

中期（Phase 2）：实现真正的增量更新——只重新嵌入变更的 chunk，支持文件删除后的向量清理，让知识库更新成本与变更量成正比。

长期（Phase 3）：异步管道成为 RAG 副驾的“神经系统”，支持实时 CDC、多模态索引、索引质量自动评测与自我修复，让知识库始终与真实世界同步。

## Open Questions

1. 单文件索引完成后是否自动通知当前在线的支持人员？（v1 不通知，仅状态可见）
2. 失败任务是否允许管理员一键重试，还是只能重新上传？（v1 建议重新上传，重试功能留给后续）
3. 是否需要为“上传即索引”提供全局开关，让管理员回到手动重建模式？（建议提供，降低变更风险）
